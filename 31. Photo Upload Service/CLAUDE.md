# Project 31: Photo Upload Service

Simple photo capture from shop floor Samsung tablet, queued for upload to ProShop entity pages. Operators tap entity type, search, snap, done. Photos queue locally and upload to ProShop via Selenium browser automation (Phase 2).

## Architecture

- Flask app on MainPC, port 5003
- Tablet connects over LAN via Fully Kiosk Browser (Android kiosk lockdown)
- Camera access via `<input type="file" capture="environment">` (works over HTTP)
- Photos resized to max 2000px, JPEG 85% quality
- SQLite tracks photo metadata and upload queue
- Background worker (Phase 2) uploads to ProShop via headless Selenium
- Three-layer QR decode: BarcodeDetector → jsQR → pyzbar server-side
- Label printing via Pillow + Brother PT-P700 print service (P30 designs ported server-side)

## Entity Types

- **workorder** — Search by WO number (flexible: "26120" matches "26-0120"), select operation, upload to written description
- **part** — Search by part number/name, then select a work order for that part (current or past), then select operation. The chosen WO becomes the entity context for the upload, which still hits the part's writtenDescription URL (same pattern as WO). QR-scan for parts skips both WO and ops selection and goes straight to capture (legacy quick-tag behavior).
- **tool** — Search by tool number/description, upload to tool page
- **equipment** — Search by number/description/location/type (flexible digit matching), upload to equipment page
- **fixture** — Search by fixture number/description
- **cots** — Search by number/description (digit-only matching, prefix-flexible). Upload goes through ProShop's "Handle New Picture" popup window, not the CKEditor base64 path — see "ProShop COTS Picture Upload" technical notes below.
- **ncr** — Search by NCR ref number/WO number/notes, upload to NCR page
- **claude** — Local-only photos saved to data/photos/claude/ (Dropbox-synced, no ProShop upload)
- **QR scan** — Scan ProShop QR label to auto-detect entity type and ID

## ProShop Written Description Upload — Technical Notes
- CKEditor dialog: name="image", tabs=[Image Info, Link, Upload, Advanced]
- Upload tab has file input inside an iframe, but no server-side upload handler (filebrowserImageUploadUrl=None)
- Working approach: insertHtml with base64 data URI, then save via fetch() interceptor
- Written description URL: `{BASE}/procnc/parts/{customer}/{partNumber}$formName=writtenDescription&opId={opNumber}`
- Customer prefix extracted from `part.proshopUrl` (e.g., `/parts/R2S1/R2S1-10020` → `R2S1`)
- Non-WO entities (equipment, tools, etc.) navigate directly to their proshop_url

## ProShop COTS Picture Upload — Technical Notes
- COTS detail pages have **no CKEditor instances** in either view or edit mode — only plain `<textarea>` fields (description, descriptionforsales, purchasingnotes, location). The base64-into-CKEditor path doesn't apply.
- Picture uploads go through a separate `window.open()` popup titled "Handle New Picture" at `/procnc/procncAdmin/fileserver/editpicture$parent_submit=true&early_parent_submit=false&hour=newTokens&page=<PARENT_UUID>&token=<CSRF>&isNew=yes`. Both `page` and `token` are per-session and embedded in the URL by ProShop's click handler.
- The "Add picture" button is wrapped in `<a id="fnXXXXXXXX">` (ProShop's "function anchor" pattern — same as on tools, parts, equipment). The click handler is bound to the anchor; a synthetic `.click()` on the inner `<button>` bubbles but doesn't satisfy the handler. **Use `ActionChains.move_to_element(anchor).click().perform()`** for a trusted-gesture click that also bypasses Chrome's popup blocker.
- Chrome must launch with `--disable-popup-blocking`; without it `window.open()` is silently suppressed for synthetic clicks and no popup appears.
- Upload flow in `_upload_cots_via_popup`: navigate to COTS page → `_checkout_page` → find anchor and ActionChains-click → `WebDriverWait(driver, 15).until(window_count_grew)` → `switch_to.window(new)` → `send_keys()` local file path on the popup's `<input type="file">` (works on hidden inputs too) → click popup SAVE CHANGES → `WebDriverWait(driver, 60).until(popup_closed)` → switch back to parent. **Don't call `_save_page` after** — the popup's SAVE CHANGES auto-commits the parent record, so the parent is already back in view mode. Trying to enumerate the parent DOM at this point can hit a renderer timeout during the parent's reload.
- COTS detail URL: `{BASE}/procnc/ots/{TYPE}/{TYPE}-{NUMBER}` — note the **prefixed** form (e.g. `/ots/PAC/PAC-223`). The bare `/ots/{TYPE}/{NUMBER}` returns a 404 page wrapped in ProShop chrome. `search_cots()` builds the prefixed form; `_upload_cots_via_popup` also normalizes defensively for any legacy queue rows still holding the bare form.
- This same popup pattern is used by tools, parts, equipment, and fixtures — same URL pattern, same anchor wrapper, just a different parent `page` UUID. As of 2026-05-17, equipment, tool, and fixture uploads all use `_upload_via_popup()` alongside COTS. Only workorder and part uploads still use the CKEditor-base64 path (they target writtenDescription forms, not picture sub-tables).

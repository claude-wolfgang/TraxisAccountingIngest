# Project 27: Accounting Ingest

Ingests vendor invoices, bills, packing slips, quotes, and purchase orders into ProShop and QuickBooks Online. Processes email attachments and scanned documents (PDF/image) via Claude AI extraction, fuzzy-matches contacts, and uploads via GraphQL mutations. Includes scan-burst-classify pipeline, tool receiving labels, and VPO receiving automation.

## Key Files

- `accounting_ingest.py` — Main script v1.4.0: ProShopClient, QBOClient, AIExtractor (burst_pdf, classify, extract, classify_html, extract_html), email body extraction, debounced vendor search, manual label printing, cert filing by VPO
- `qbo_auth.py` — QBO OAuth 2.0 production token exchange (manual paste-URL flow via Intuit Playground redirect)
- `tool_receiving_labels.py` — Tool receiving label generator: VPO→tool library lookup (two-client), Brother PT-P700 label printing via P22 print service
- `scan_relay.py` — Watches Pictures folder, moves stable PDFs to Scanned folder
- `proshop_fields.py` — ProShop schema introspection utilities
- `diag_purchase_order.py` — Diagnostic for addPurchaseOrder mutation permissions
- `proshop_basic_auth_test.py` / `proshop_basic_auth_test2.py` — Basic auth (`/api/beginsession`) probes proving `addCustomerPo` works under user-credential flow where OAuth+`auth_010` was blocked
- `inspect_po_page.py` — Visible-mode Chrome DOM inspector for ProShop PO pages (mirrors P31's `inspect_user_page.py`). Takes `--url`, walks default + candidate sub-forms + sidebar `$formName=` links, dumps CKEditor count, file inputs, visible buttons. Outputs JSON + PNG to `logs/`. Verdict 2026-05-07 against VPO 263104: read mode has no CKEditor and no file inputs anywhere (same dead-end as user pages); attach surface is gated behind a `Checkout` button. Phase-2 (click-Checkout-then-probe) not yet implemented.

## Manual Handling reminder

Anything classified as `MANUAL_HANDLING` (vendor credit memos, debit memos, adjustment notes, chargebacks) lands in `Accounting Inbox/Manual Handling/`. The Tkinter `on_close` handler in `accounting_ingest.py` sweeps that folder when the GUI window is closed: if non-empty, a modal `messagebox.showinfo` lists the files and `os.startfile(MANUAL_HANDLING_FOLDER)` pops Explorer so Wolfgang sees the backlog before walking away. No auto-deletion — files stay until manually removed.

Auto-opens in Explorer when the GUI closes if non-empty; **`Accounting Inbox/Filed/UTILITY_NOTIFICATION/`** — non-bill utility usage notifications (HTML body + JSON sidecar); **`Accounting Inbox/Filed/PAYMENT_VOUCHER/`**; **`PROSHOP_WRITES_AUDIT.md`** — full audit (under P27 folder) of every ProShop write the ingest has made, pulled from `ingest_queue.db` (status=UPLOADED + UPLOAD_FAILED). Events: `BILL_CREATED` / `BILL_FAILED` / `PDF_ATTACHED` / `PDF_ATTACH_FAILED` with UTC ISO timestamp, env, vendor+id, account+id, account_source (`vendor_last_bill` or `global_fallback`), amount, doc_number, qbo_url, error.
Consumes: ProShop GraphQL API (ACCOUNTING_CLIENT_ID for purchaseorders/customerpos scope, TOOLKIOSK_CLIENT_ID for tools scope; **basic-auth path uses `ACCOUNTING_SCOPE` env var which gained `workorders:rwdp` on 2026-05-18** to support WorkOrder reads + status mutations for the back-feed mechanism), QBO REST API (OAuth 2.0 production, refresh token auto-renews — production token gets stale if env stays in sandbox mode for >100 days; `_complete_qbo_auth.py` is a one-shot helper that takes a redirect URL on argv and writes the fresh refresh token + realm to `.traxis.env`, bypassing the interactive paste prompt in `qbo_auth.py`), Microsoft Graph API (email attachments + email body text via get_body()), Claude AI API (Sonnet for extraction, Haiku for HTML classification), P22 print service (10.1.1.242:5002 /api/print-image), .traxis.env (credentials), Scanner → Pictures folder (scan_relay.py)
**Reconciliation toolkit (read-only, added 2026-05-18):** `read_proshop_invoices.py` (Stage 2 — matches ProShop invoices ↔ QBO by `invoiceId`↔`DocNumber` + total; 30-day window), `read_proshop_unbilled_shipments.py` (Stage 1 — shipped PackingSlips with blank `invoicePlainText`; 30-day window; `$` totals NOT shown because `pricePer` is null on un-invoiced slips), `read_payment_reconcile.py` (Stage 3 — QBO `Invoice.Balance` vs ProShop `Invoice.status`; 180-day window; flags `STOP_CHASING` / `PARTIAL` / `UNEXPECTED_OWED` / `AGREED` / `NOT_IN_QBO`). All three write JSON sidecars to `27. Accounting Ingest/logs/`. Invoked individually OR together via the `/reconcile` slash command at `.claude/commands/reconcile.md` (root project) which runs Stage 1+2 in parallel and formats output as a punch list referencing memories `project_proshop_qbo_zero_dollar_set1`, `project_customer_code_mapping`, `project_proshop_invoice_schema_gotchas`. `logs/proshop_wo_flip.log` — tab-delimited audit of any WorkOrder status mutations (currently manual, will be auto-fed by `flip_wo_invoiced_from_qbo.py` once built). `WO_STATUS_FLIP` and `WO_STATUS_ROLLBACK` events with UTC ISO timestamp, WO#, part, old/new status.

Contracts: PO line items use `toolNumber` field (not `itemNumber`) for tools and COTS. `orderNumber` field carries brand + EDP. `shipTo` defaults to Traxis MFG 511 E St Elmo Rd. updatePurchaseOrder mutation: `id` is a separate arg (not inside `data`), poItems use selector {field: "orderNumber", value: "..."} + data {receivedQty, releasedQty, receivedDate, releasedDate}. P22 print service contract: POST /api/print-image with {image_base64, copies, label_name}. Customer PO mutations: OAuth path historically blocked at `acceptNewRecord` permission (root cause unknown); basic-auth flow (`/api/beginsession` with PROSHOP_USERNAME=full-email + PROSHOP_PASSWORD + space-delimited scope) bypasses this and works for `addCustomerPo`. Service user `auth_010` deleted 2026-05-06; AccountingConnector OAuth client is now mapped to nothing/undefined. **Customer PO writes now go through inline `_BasicAuthSession`** (`add_customer_po`, `update_customer_po`) — beginsession → query-param token → auto-refresh on 401. Every other ProShop mutation stays on OAuth. Mirrors P35's `purchasing/proshop_basic_auth.py`; kept inline to avoid cross-project Python import. ProShop audit log shows Wolfgang's name as creator for these mutations, not a service user. **QBO Bill account categorization mirrors the QBO UI**: `create_bill` resolves the line-item AccountRef by querying the vendor's most recent Bill (`SELECT * FROM Bill WHERE VendorRef = ? ORDERBY TxnDate DESC MAXRESULTS 1`) and reusing line[0]'s account; falls back to first COGS, then first Expense, if the vendor has no prior Bills. Source recorded in qbo_audit.log per push. **ProShop dup-check** uses per-doc-type filter field (`bills.referenceNumber`, `customerPOs.clientPONumber`, `purchaseOrders.confirmationNumber`, `packingSlips.referenceNumber`, `quotes.referenceNumber`); normalizes leading `PO`/`P.O.`/`PO#`/`INV` prefix; parses dates loose (ISO or `M/D/YYYY`); 6-month window. On match: hard refuse — no override — row marked `POSSIBLE_DUPLICATE` with `proshop_url` pointing at the existing record. **Permanent reject**: `check_local_duplicate` prefers REJECTED matches over any other; re-ingested content matching a previously-rejected `doc_hash` auto-rejects on insert (applies to email PDF, email body, and scan paths). **CPO confirmation auto-generation**: after successful `addCustomerPo`, background thread runs `download_proshop_confirmation()` — headless Chrome logs in, navigates to `{proshop_url}?formName=confirmation&printerfriendly=yes`, prints to PDF via Chrome DevTools `Page.printToPDF`, saves to `Outgoing Customer Confirmations/`. Then `updateCustomerPo` flips `confirmationSent=true` + `confirmationSentBy` + `confirmationNotes=<file path>` on the ProShop record. **`MANUAL_HANDLING` doc type** files to `Accounting Inbox/Manual Handling/` (top-level, visible). The Tkinter `on_close` handler pops a modal + opens Explorer if the folder is non-empty — never auto-deletes. **ProShop Invoice number is `Invoice.invoiceId`** (not `legacyId`, no `invoicePlainText` on Invoice itself). **`Invoice.status` is effectively unused** — binary Outstanding/Booked with only 2 Booked across 1378 invoices over 5 years; the real "is this invoiced" status lives on **`WorkOrder.status`** (values: Active, Manufacturing Complete, Shipped, Invoiced, Complete, Canceled). **`ProShop Contact.accountingGuid`** is the link field for QBO sync — despite the "GUID" label it takes the **QBO Customer DisplayName** (e.g. `OPMobility`), NOT the numeric Id. Set this on the Contact record to clear "Bill Sync Error" banners on that customer's invoices. **`updateWorkOrder` mutation confirmed working under basic auth 2026-05-18** (probed on OPM1 26-0155 + batch of 11 stuck WOs). **ProShop→QBO sync is create-only** — once an invoice exists in QBO, ProShop-side edits do NOT propagate; must repeat in QBO manually or Stage 2 reconcile will flag as amount mismatch.

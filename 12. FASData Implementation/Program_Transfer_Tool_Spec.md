**Version:** 1.0 Draft

**Author:** Claude Code (investigation & research)

## 1. Executive Summary

Traxis Manufacturing currently uses the **Fanuc Program Transfer Tool (PTT)** for transferring NC programs to Fanuc CNC machines and a manual **SSH/SCP workflow** (via Raspberry Pi Zero) for the Haas VF-5/40. This creates a fragmented operator experience — two different tools, two different workflows, no audit trail.

This document specifies a **unified CNC program transfer tool** that:
- Replaces/complements Fanuc PTT with a single, operator-friendly interface
- Abstracts away connection differences (FOCAS/FTP, SSH/SCP, SMB, RS-232)
- Provides transfer logging, version awareness, and safety confirmations
- Deploys as a standalone Windows .exe requiring no admin rights

The goal is **one tool, one workflow, any machine**.

## 2. Shop Environment Summary

### 2.1 NC Program Storage Locations

| Location | Purpose | Naming Convention |
|----------|---------|-------------------|
| `D:\Users\MainPC\Documents\NC Files For Transfer\` | **Primary transfer staging folder** — programs staged here before sending to machines | Subfolders by job number (e.g., `26-0024\`) containing `O####.nc` files |
| `D:\Dropbox\NC Programs\{PartNumber}\` | **TPM output** — Fusion 360 TraxisProgramManager post-processes files here | `{PartNumber}_OP{XX}_v{N}.nc` (e.g., `UTA1-NP000674_OP60.nc`) |
| `C:\FASData\` | **Machine monitoring databases** — NOT program files | `monitoring.db`, `tooling.db`, `shophub.db` |
| `D:\Dropbox\OPERATIONS Traxis\MACHINES\` | **Machine documentation** — manuals, backups, parameter files | Mixed formats |

### 2.2 File Formats Found

| Extension | Format | Prevalence |
|-----------|--------|------------|
| `.nc` | Fanuc G-code with O-number header | Primary (all machines) |
| `.par` | Haas parameter backup | Haas only |
| `.SET` | Haas settings backup | Haas only |

All NC program files are **ASCII G-code** with Fanuc O-number format:
```
%
O1234 (PART-NAME-OP60)
G90 G54 G40
...
M30
%
```

### 2.3 Naming Conventions

Two naming systems coexist:

1. **Fanuc O-number** (legacy): `O0100.nc`, `O0101.nc` — used in the `NC Files For Transfer` folder, organized by job number subfolder
2. **TPM naming** (new standard): `{PartNumber}_OP{XX}_v{N}.nc` — generated by TraxisProgramManager from Fusion 360 CAM

The O-number inside the file content is what the Fanuc CNC registers as the program number, regardless of the Windows filename.

### 2.4 Current Workflow (How Programs Get to Machines)

```
Fusion 360 CAM
    │
    ├──[TPM Post]──► D:\Dropbox\NC Programs\{PartNumber}\{PartNumber}_OP60_v1.nc
    │
    ├──[Manual Copy]──► D:\Users\MainPC\Documents\NC Files For Transfer\{JobNumber}\O####.nc
    │
    ▼
Fanuc PTT (GUI drag-and-drop)──► Fanuc CNC (FOCAS, port 8193)
    OR
SSH/SCP (3-step manual commands)──► Pi Zero──► Haas USB
```

**Pain points:**
- Two separate tools for two machine brands
- No audit trail of what was sent where
- Manual SSH commands for Haas — error-prone, intimidating for operators
- No version awareness — easy to send wrong revision
- Fanuc PTT cannot be scripted or automated

## 4. Connection Protocol Details

### 4.1 Fanuc FOCAS2 (Primary — Confirmed for All Fanuc Machines)

| Parameter | Value |
|-----------|-------|
| **Protocol** | FOCAS2 (proprietary binary over TCP) |
| **Port** | 8193 |
| **Authentication** | None (handle-based) |
| **Python library** | `ctypes` + `Fwlib32.dll` (already in project), or `chattertools` (`pip install chattertools`) |
| **Transfer direction** | Bidirectional (PC-initiated push AND pull) |
| **Additional capabilities** | Machine status, program directory, tool/work offsets, drip feed |
| **Confirmed working on** | T2 (10.1.1.82), M2 (10.1.1.159), M3 (10.1.1.118), M6 (10.1.1.106), M8 (10.1.1.202) |

**Why FOCAS, not FTP:** Tested 2026-03-18 on M3. The Fanuc 0i-MF Embedded Ethernet only supports FTP as a **client** (CNC pulls from PC). There is no FTP **server** — that requires an optional Data Server board which these machines don't have. FOCAS is the only PC-initiated push protocol available. This is the same protocol Fanuc PTT uses internally.

**Send program to CNC sequence (PC → CNC):**
```
cnc_allclibhndl3(ip, port, timeout, &handle)  # Connect
cnc_dwnstart3(handle, program_number)           # Begin download
cnc_download3(handle, &length, data)             # Send chunks (loop)
cnc_dwnend3(handle)                              # End download
cnc_freelibhndl(handle)                          # Disconnect
```

**Receive program from CNC sequence (CNC → PC):**
```
cnc_allclibhndl3(ip, port, timeout, &handle)  # Connect
cnc_upstart3(handle, program_number)            # Begin upload
cnc_upload3(handle, &length, data)              # Receive chunks (loop)
cnc_upend3(handle)                              # End upload
cnc_freelibhndl(handle)                         # Disconnect
```

**List programs on CNC:**
```
cnc_rdprogdir2(handle, type, &start, &count, &progdir)
```

**Existing assets:**
- `Fwlib32.dll` + supporting DLLs already in `focasmonitor\` directory
- `Focas.cs` wrapper has upload functions (`cnc_upstart`, `cnc_upload`, `cnc_upend`) and `cnc_rdprogdir2`
- Download functions (`cnc_dwnstart3`, `cnc_download3`, `cnc_dwnend3`) need Python ctypes bindings

**Gotchas:**
- Only ONE FOCAS connection per CNC at a time — **will conflict with FocasMonitor** if both connect simultaneously
- Handle leaks cause CNC to refuse connections until reboot — always use `try/finally` with `cnc_freelibhndl`
- Download overwrites existing programs with same O-number silently
- `Fwlib32.dll` is Windows-only (32-bit) — need 32-bit Python or ctypes bridge
- Struct alignment: use `_pack_ = 4` in ctypes structures
- CNC must be in EDIT mode for program transfer on some models

### 4.2 Fanuc FTP (Not Available — Documented for Reference)

**Status: RULED OUT** — tested 2026-03-18 on M3 (Fanuc 0i-MF).

The Embedded Ethernet FTP on these controls is **client-only** (I/O Channel = 9). The CNC connects out to a PC-hosted FTP server to pull files. There is no FTP server on the CNC for PC-initiated push. An FTP server requires the optional **Data Server board** (separate hardware with its own Ethernet port and IP), which is not installed on any Traxis Fanuc machine.

The FTP client settings screen (SYSTEM → ETHER → FTP TRANS) was found pre-configured on M3 pointing to `10.1.1.165:21` with user `ftp` — indicating FTP-pull has been used historically, but is operator-initiated from the CNC side, not suitable for our tool.

### 4.3 Haas CHC via Pi Zero (SSH/SCP)

| Parameter | Value |
|-----------|-------|
| **Protocol** | SSH/SCP (port 22) |
| **Pi Zero IP** | 10.1.1.149 |
| **Username** | `haasmill1` |
| **Machine IP** | 10.1.1.220 (Haas directly — not used for transfer) |
| **Python library** | `paramiko` + `scp` (`pip install paramiko scp`) |
| **Transfer direction** | Bidirectional (SCP both ways) |
| **Overhead** | 3-5 seconds for mount/unmount cycle |

**Three-step transfer sequence:**
1. `ssh haasmill1@10.1.1.149 "/home/haasmill1/pre-copy.sh"` — disconnect USB from Haas, mount for writing
2. `scp "file.nc" haasmill1@10.1.1.149:/mnt/usb_share/` — copy file
3. `ssh haasmill1@10.1.1.149 "/home/haasmill1/post-copy.sh"` — unmount, re-attach USB to Haas

**Post-transfer:** Operator must go to Haas control → LIST PROGRAM → USB → select and load file.

**Gotchas:**
- USB gadget and Pi filesystem CANNOT access disk image simultaneously — violating this causes filesystem corruption
- FAT32 has no journaling — power loss during write corrupts the image
- If any step fails, ALWAYS run post-copy.sh to re-attach USB
- Use SSH key auth instead of passwords for reliability
- Pi Zero can be slow after boot — implement retry logic (2-3 attempts)

### 4.4 Haas NGC (Future — SMB)

| Parameter | Value |
|-----------|-------|
| **Protocol** | SMB 2.0 (Net Share) |
| **Port** | 445 |
| **Username** | `haas` (fixed, cannot change) |
| **Password** | User-defined on CNC |
| **Share** | `\\<cnc-ip>\User Data` (read/write NC programs) |
| **Python library** | `smbprotocol` (`pip install smbprotocol`) |

**Python implementation:**
```python
import smbclient

smbclient.register_session("10.1.1.xxx", username="haas", password="...")
with open("local.nc", "rb") as lf:
    with smbclient.open_file(r"\\10.1.1.xxx\User Data\program.nc", mode="wb") as rf:
        rf.write(lf.read())
```

**Gotchas:**
- Haas defaults to SMB 1.0 — change to SMB 2.0 (SMB 1.0 disabled on Windows 10/11)
- Domain name max 15 characters
- CNC locks programs it is executing — cannot overwrite running program
- No SSH/SFTP — SMB is the only file transfer protocol on NGC

### 4.5 RS-232 Serial (Legacy Fallback)

| Parameter | Fanuc (legacy) | Haas (pre-2016) |
|-----------|---------------|-----------------|
| **Baud Rate** | 9600 | 9600 |
| **Data Bits** | 7 | 7 |
| **Parity** | Even | Even |
| **Stop Bits** | 2 | 1 or 2 |
| **Flow Control** | XON/XOFF | XON/XOFF |
| **Python library** | `pyserial` (`pip install pyserial`) |

**Speed:** ~52 KB/min at 9600 baud — a 100KB program takes ~2 minutes.

**Gotchas:**
- Use FTDI-based USB-to-serial adapters (Prolific PL2303 has Windows driver issues)
- CNC serial cables use null-modem wiring (TX/RX swapped)
- 7 data bits, NOT 8 — wrong setting = garbled data
- No error detection (no CRC/checksum) — corruption is silent
- Programs must start and end with `%`

| Feature | Priority | Rationale |
|---------|----------|-----------|
| Integration with NC Files For Transfer folder | **Must Have** | This is where programs live today |

| Recent files list | **Must Have** | Operators often re-send the same programs |
| Favorites / pinned programs | **Nice to Have** | Low effort, nice for repeat jobs |

#### 5.3.1 Smart Folder Resolution (Detail)

The tool should intelligently suggest the right folder for the selected machine, using context from the existing Traxis ecosystem. The resolution chain:

```
1. ProShop Query (highest priority)
   │  "What WO is active on Mill-6 right now?"
   │   → WO 26-0024 → Part Number UTA1-NP000674
   │
   ├──► Look in: D:\Dropbox\NC Programs\UTA1-NP000674\
   │    (TPM output — latest versions with {PartNumber}_OP{XX}_v{N}.nc naming)
   │
   └──► Look in: D:\...\NC Files For Transfer\26-0024\
        (Legacy staging folder — O-number named files)

2. Remembered Folder (fallback)
   │  "Last time Mill-6 was used, the operator was browsing this folder"
   │   → Persist in local config (SQLite or machines.json)
   │
   └──► Show that folder, but with a banner: "Showing last-used folder"

3. Default Root (last resort)
   └──► Show NC Files For Transfer root, let operator navigate
```

**Data sources for the guess:**
- **ProShop GraphQL API** — `workOrders` query filtered by work cell + active status → returns WO number + part number
- **ProShopBridge attributes** — If the operator has Fusion open, `ProShopBridge:PartNumber` tells us the active part
- **TPM naming convention** — `{PartNumber}_OP{XX}_v{N}.nc` — tool can parse the latest version automatically
- **Folder scan** — Match WO number subfolders in `NC Files For Transfer` to active WO

**Key behavior:**
- The suggestion is just that — a suggestion. Operator can always browse elsewhere.
- When the operator manually picks a different folder, the tool remembers it for next time on that machine.
- If ProShop is unreachable, silently fall back to remembered folder (no error popup).

## 7. Recommended Technology Stack

### 7.1 Recommendation: Python + CustomTkinter

| Criterion | Choice | Rationale |
|-----------|--------|-----------|
| **Language** | Python 3.11+ | Already used across all Traxis automation (TPM, Tool Kiosk, monitoring scripts) |
| **GUI Framework** | CustomTkinter | Modern dark-mode look, minimal dependencies, easy PyInstaller packaging |
| **Packaging** | PyInstaller → single .exe | No install needed, no admin rights, operator double-clicks and it runs |
| **Transfer: Fanuc FOCAS** | `ctypes` + `Fwlib32.dll` | Primary Fanuc protocol — DLLs already in project, stdlib ctypes |
| **Transfer: Haas CHC** | `paramiko` + `scp` | Well-tested SSH/SCP libraries |
| **Transfer: Haas NGC** | `smbprotocol` | Pure Python SMB2/3 client |
| **Transfer: RS-232** | `pyserial` | Standard serial library |
| **Audit Log** | SQLite via `sqlite3` (stdlib) | Consistent with existing tools |
| **Config** | `machines.json` | Same format as FocasMonitor — can share/extend |

### 7.2 Why CustomTkinter over Alternatives

| Option | Verdict | Reason |
|--------|---------|--------|
| **CustomTkinter** | **Selected** | Fastest to prototype, single .exe (~15-30MB), modern look, matches project complexity |
| Flask web app | Runner-up | Good if multi-PC browser access needed; more complex deployment (server process) |
| PySide6 (Qt) | Upgrade path | If CustomTkinter hits limits on tables/trees/touch; larger package (~80-150MB) |
| Electron | Overkill | 100MB+ installer, bundles Chromium, unnecessary for this scope |
| .NET WPF | Wrong stack | C# — different from the Python ecosystem already in place |

### 7.3 Dependency Summary

```
# requirements.txt
customtkinter>=5.2

scp>=0.14              # SCP file transfer over paramiko

# No pip install needed for:
# - ctypes (stdlib) — Fanuc FOCAS DLL loading (Fwlib32.dll)
# - sqlite3 (stdlib) — Audit log database
```

### 7.4 Recommended Architecture

```
TraxisTransfer/
├── main.py                 # App entry point + CustomTkinter UI
├── config.py               # Load machines.json, app settings
├── machines.json            # Per-workstation machine definitions
├── transfer_log.db          # SQLite audit trail (auto-created)
├── drivers/
│   ├── base.py             # Abstract driver interface (send/receive/list/status)
│   ├── fanuc_focas.py      # FOCAS via ctypes + Fwlib32.dll (primary Fanuc driver)
│   ├── haas_chc_ssh.py     # SSH/SCP via paramiko (Pi Zero path)
│   ├── haas_ngc_smb.py     # SMB via smbprotocol
│   └── serial_dnc.py       # RS-232 via pyserial (legacy)
├── folder_resolver.py       # Smart folder association (ProShop lookup + remembered folders)
├── proshop_client.py        # Lightweight GraphQL client (active WO → part number lookup)
├── models/
│   ├── machine.py          # Machine config dataclass
│   └── transfer.py         # Transfer job with logging
└── Fwlib32.dll             # FOCAS DLLs (required for Fanuc transfer)
```

`folder_resolver.py` implements the smart folder chain:
1. Query ProShop for active WO on this machine's work cell (`proshop_pot_id` from machines.json)
2. Map WO → part number → scan `NC Programs\{PartNumber}\` and `NC Files For Transfer\{WO}\`
3. Merge both folder contents into a single file list (TPM files at top, staging files below)
4. Fall back to last-remembered folder if ProShop is unreachable
5. Persist operator's manual folder choice per-machine in local config

### 7.5 Implementation Priority

1. **Fanuc FOCAS driver** — covers 5 active Fanuc machines; uses existing Fwlib32.dll + ctypes
2. **Haas CHC SSH/SCP driver** — wraps existing Pi Zero scripts via paramiko
3. **Core UI** — file browser, smart folder resolver, machine selector, transfer button, log
4. **Smart folder resolution** — ProShop WO lookup + remembered folders per machine
5. **Audit logging** — SQLite transfer log
6. **Haas NGC SMB driver** — ready for future NGC machines
7. **RS-232 serial driver** — only if specific machines require it

## 9. Risk List

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| ~~FTP not enabled on Fanuc CNCs~~ | — | — | ✅ **Resolved:** FTP ruled out. FOCAS confirmed as only PC-initiated Fanuc protocol. |
| **FOCAS connection conflict with FocasMonitor** | **High** | **High** — transfer will fail if monitor holds connection | Implement connection coordination: transfer tool signals FocasMonitor to release, waits, connects, transfers, signals done. Or use short connect/transfer/disconnect cycles between monitor poll intervals (currently 60s). |
| **Pi Zero filesystem corruption** | Low | High — Haas loses USB drive access | Implement atomic pre/post-copy with error recovery; MD5 verification |
| **Wrong program sent to wrong machine** | Low (with UI) | Critical — crash/scrap | Confirmation dialog with machine name + program details; per-workstation config limits target machines |
| **Network interruption during transfer** | Low | Medium — partial transfer | Implement timeout + retry; verify transfer completion; cleanup on failure |
| **Operator bypasses tool and uses PTT** | Medium | Low — no audit trail for that transfer | Training; make the new tool easier than PTT so operators prefer it |
| **CustomTkinter limitations** | Low | Medium — may need to switch to PySide6 | The UI is simple enough; CustomTkinter handles it. PySide6 is the upgrade path. |
| **Fwlib32.dll is 32-bit only** | **Medium** | **High** — requires 32-bit Python or ctypes bridge | FOCAS is now the primary Fanuc driver, so this must be solved. Options: use 32-bit Python for the whole app, or use a 32-bit subprocess/bridge for FOCAS calls. |
| **RS-232 USB adapter driver issues** | Medium | Low — only affects legacy path | Use FTDI-based adapters; document known-good models |
| **Large file transfer failures** | Low | Medium — PTT has known issues with >185KB files | FOCAS chunked transfer handles large files; test with largest programs in shop |

## 10. Appendix: Fanuc PTT Limitations (Why Replace It)

| Limitation | Details |
|-----------|---------|
| **Not scriptable** | No CLI, no API, no COM automation. GUI-only. |
| **Windows-only** | Cannot be used from Linux or automated systems |
| **O-number confusion** | Uses O-number from file content, ignores Windows filename entirely |
| **No audit trail** | No log of what was sent where or by whom |
| **No Haas support** | Fanuc-only — cannot talk to Haas machines |
| **No RS-232** | Ethernet FOCAS only |
| **Large file failures** | Reported errors on files >185 KB |
| **Single-brand** | Only works with Fanuc CNCs |

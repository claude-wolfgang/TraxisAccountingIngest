## Project Overview

### What Is This?
A natural language interface for querying ProShop ERP. Instead of clicking through the ProShop web UI, users type (or speak) questions like:
- "What's the status of WO 25-0001?"
- "Show me all open work orders"
- "What operations are on part TRA1-TEMP?"

### Why Build It?
- Faster access to info on shop floor (handhelds, phones)
- Reduce time walking to computers to look things up
- Leverage the ProShop API we're already paying $3,630/year for

### Who Would Use It?
| Person | Typical Questions |
|--------|-------------------|
| Chris (setup/inspection) | "What's the next job?" / "What ops are on this part?" |
| Jose (operator) | "Is this WO ready?" / "What's the quantity?" |
| Rene (purchasing/ops) | "What's late?" / "R2Sonic orders status?" |
| Wolfgang/Garrett | Everything |

## Technical Architecture

### Architecture: v1 (Regex-Based, January 2026)
```
User Question → Regex Pattern Match → GraphQL Template → ProShop API → ASCII Table
```
- 60 regex patterns, no LLM, no conversation memory
- Accuracy: 68% on test suite (failed on typos, slang, indirect phrasing)

### Architecture: v2 (Claude-Powered, March 2026)
```
User Question
    ↓
1. CLAUDE INTENT CLASSIFIER (Haiku, tool-use)
   - Understands natural language, typos, slang
   - Picks correct query tool from 18 options
   - Resolves pronouns from conversation history ("it", "that job")
    ↓
2. CONVERSATION MEMORY
   - Tracks last 20 turns
   - Enables follow-up questions
    ↓
3. PROSHOP GraphQL API
   - OAuth 2.0, caching, rate limiting
    ↓
4. CLAUDE RESPONSE FORMATTER (Haiku)
   - Converts JSON to natural language
   - Flags urgent items (late orders, data issues)
   - Markdown-formatted, scannable output
    ↓
Natural Language Answer
```
- Accuracy: 100% on same test suite
- Avg response time: ~3.3s (1.3s classify + 0.4s ProShop + 1.6s format)
- Cost: ~$0.002 per query (~$0.20/day at 100 queries)

## Known Issues & Limitations

### Fixed in v2 (March 2026)
- [x] Rate limiting (1 sec between API calls)
- [x] 12-month default lookback filter
- [x] Intent classification failures on typos/slang (Claude handles these now)
- [x] No conversation context (conversation memory added)
- [x] Rigid ASCII table output (natural language responses now)

### Remaining Issues

1. **Fixtures scope not available**
   - `fixtures:r` scope needed but couldn't be added to API application
   - ProShop admin scope editor is finicky - changes don't always save
   - Workaround: None currently. Need ProShop support or admin retry.

2. **No server-side filtering**
   - ProShop API doesn't support status/date filtering server-side
   - Must fetch all records + filter client-side
   - Slow queries (status, late, count) fetch up to 2500 records

3. **Claude response time variability**
   - Haiku typically responds in 1-2s but can spike to 10-15s under load
   - Total query time ranges from 3s (fast) to 30s (worst case)

### ProShop API Quirks
- Written Description API bug still exists (use Selenium workaround for writes)
- Parts filter sometimes returns empty for known parts
- WO nested filter by PO number doesn't work (fetch all, match client-side)
- setupTime/runTime returned in SECONDS not hours

## Decisions Made

| Decision | Rationale |
|----------|-----------|
| Read-only first | No risk of corrupting data while learning API |
| CLI before web | Faster to prototype, validate queries work |
| Pattern matching for intent (v1) | Simpler than LLM, no extra API cost |
| **Claude tool-use for intent (v2)** | **100% vs 68% accuracy, handles real-world input** |
| **Claude Haiku for responses (v2)** | **Natural language, flags issues, ~$0.002/query** |
| **Conversation memory (v2)** | **Follow-up questions just work ("what about its ops?")** |
| 12 month lookback default | Keep results relevant, reduce API load |
| Rate limiting | Prevent hogging ProShop during normal use |
